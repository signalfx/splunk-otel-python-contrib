How Telemetry Gets from Your App to Splunk O11y

The journey of a span — from code to dashboard

1. What Is a Span?

A span is a record of something that happened. Every LLM call, every tool call, every security check creates a span. It's just a data structure:

Your app doesn't build these by hand. The instrumentors create them automatically when LangChain/OpenAI/AI Defense functions run. They're like invisible recorders.

2. The Full Journey of a Span

① Your Code Runs
agent.invoke("Find flights to Tokyo") → calls GPT-4o-mini → gets response
This is your business logic. It works the same with or without telemetry.
▼
② Instrumentor Creates Spans (invisible to your code)
LangchainInstrumentor saw the agent.invoke() call → created "invoke_agent flight_specialist" span
Also created "chat gpt-4o-mini" span for the LLM call, with token counts, model name, etc.
▼
③ BatchSpanProcessor Collects Spans
Spans don't get sent one by one. They're batched — collected into groups and sent periodically.
At the end, provider.force_flush() sends any remaining spans.
▼
④a ConsoleSpanExporter
Prints span JSON to stdout
(for debugging in pod logs)
No network call — just print()
④b OTLPSpanExporter
Sends spans via gRPC to the collector
Endpoint: ai-defense-collector:4317
Network call — gRPC POST with protobuf
▼
⑤ Dedicated OTel Collector
Adds: k8s.cluster.name, deployment.environment
Generates spanmetrics (RED metrics from spans)
Forwards to lab0 + rc0 (different token per realm)

Key insight: Your app never modifies LLM responses before sending telemetry. The instrumentors just observe what happened (function name, duration, token counts, model, etc.) and create spans from those observations. The actual LLM response goes to your business logic unchanged.

3. Two Ways to Set Up the Exporter

The exporter is what sends spans from your app to the collector. There are two ways to set it up:

Manual Setup What our demo does

You write the setup code in your app:

from opentelemetry import trace from opentelemetry.sdk.trace import TracerProvider from opentelemetry.sdk.trace.export import ( BatchSpanProcessor, ConsoleSpanExporter, ) from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import ( OTLPSpanExporter, ) from opentelemetry.sdk.resources import Resource # Create provider with service name resource = Resource.create({ "service.name": "my-app", }) provider = TracerProvider(resource=resource) # Add exporters provider.add_span_processor( BatchSpanProcessor(ConsoleSpanExporter()) ) provider.add_span_processor( BatchSpanProcessor(OTLPSpanExporter( endpoint="http://collector:4317", insecure=True, )) ) trace.set_tracer_provider(provider)

~15 lines of boilerplate in your app

✓ Can add ConsoleExporter for debugging

✓ Full control over configuration

✗ Code mixed with business logic

Zero-Code Setup Recommended

No setup code in your app. Everything via env vars:

# Your app code — NO OTel setup at all # Just your business logic from langchain_openai import ChatOpenAI llm = ChatOpenAI(model="gpt-4o-mini") result = llm.invoke("Plan a trip")

Run with the opentelemetry-instrument wrapper:

# Set env vars (or put in K8s configmap) export OTEL_SERVICE_NAME="my-app" export OTEL_EXPORTER_OTLP_ENDPOINT="http://collector:4317" export OTEL_TRACES_EXPORTER="otlp" export OTEL_METRICS_EXPORTER="otlp" export OTEL_LOGS_EXPORTER="otlp" # Run your app with the wrapper opentelemetry-instrument python main.py

0 lines of OTel code in your app

✓ Clean separation — config vs code

✓ Easy to change without rebuilding

✗ No ConsoleExporter (less debugging)

4. What the Collector Does

The collector sits between your app and Splunk. It receives spans, enriches them, and forwards to the right destinations.

The collector does NOT modify span content — it adds resource attributes (cluster name, environment), generates additional metrics from spans (via spanmetrics connector), and routes data to the right destinations with the right auth tokens. Your span data passes through untouched.
